# 『歩継ぎ（あゆつぎ）』プロジェクト

## 概要

歩数で一族を存続させるAndroidアプリ。週間目標歩数を達成し続ける限り一族が代をつなぎ、未達なら家系断絶（滅亡）する。Health Connectから歩数を取得するサーバーレス（クラウド連携なし）構成。

## 最重要ルール：仕様書が正

- 仕様の正本は `specs/` の4部。**数値・用語・判定ルールは必ず仕様書を読んで確認すること。記憶や推測で書かない。**
  - `specs/01_ゲームルール仕様.md` … 週間目標・週次判定・滅亡・SUSPENDED・グレース週
  - `specs/02_人生シミュレーション仕様.md` … 一代の長さ・ステージ・寿命計算・遺伝・家宝
  - `specs/03_人生イベント表.md` … イベント発生制御・イベント定義・世代伏線
  - `specs/04_画面仕様.md` … 5画面構成・ナビゲーション・演出
- 技術設計の正本は `docs/design/`（技術設計フェーズで作成）
- 実装中に仕様の解釈・決定が必要になったら、**先に仕様書または設計書に追記してからコードを書く**。「コードだけが知っている仕様」を作らない
- 仕様書間で記述が矛盾する場合は作業を止めてユーザーに報告する

## 絶対に守る仕様の不変条件（詳細は必ず仕様書で確認）

- 週の定義：月曜0:00:00〜日曜23:59:59、締めは日曜24時固定（specs/01 §0-1）
- 歩数取得失敗時：再取得猶予72時間＋最終フォールバック7日、保留中はSUSPENDED（specs/01 §0-2）
- 週間目標：下限7,000／上限210,000／刻み1,000歩（specs/01 §0-4）
- 初週はグレース週（滅亡判定対象外）、再出発時はグレースなし（specs/01 §0-6）
- 標準寿命20日・固定ステージ合計13日・老年期＝寿命−13（specs/02主管）
- 歩数→ステータス変換・寿命計算は**目標相対**（絶対歩数ではない）（specs/02）
- エクスポート/インポートはMVPに含む（specs/01 §0-3）

## 技術スタック

- Kotlin / Jetpack Compose / Material 3
- Health Connect API（歩数取得）
- Room（ローカル永続化）／ WorkManager（バックグラウンド処理）
- サーバーレス：外部サーバー・クラウドへの通信は実装しない
- **iOS展開を見据えた分離（方針B）**：将来のKotlin Multiplatform化に備え、ドメインロジック（週次判定・人生シミュレーション・イベント発生制御）はAndroid非依存の純Kotlinモジュールに分離する。ドメイン層に `android.*` への依存を持ち込まない。MVPはAndroid専用（iOS対応作業はしない）
- その他のアーキテクチャ決定は `docs/design/D01` に従う

## 開発ルール

- 判定ロジック（週次判定・寿命・イベント発生）は**テスト駆動**：テストを先に書く
- 時刻を扱うコードは必ずタイムゾーン・週境界（月曜0:00）・端末時刻変更を考慮し、テストに境界値ケースを含める
- 現在時刻の直接取得（`System.currentTimeMillis()`等）をドメイン層に書かない。Clock抽象を注入してテスト可能にする
- コミットは機能の区切りごとに小さく。コミットメッセージは日本語可
- UIテキストは日本語。コードのコメントも日本語可

## よく使うコマンド

```bash
./gradlew build          # ビルド
./gradlew test           # ユニットテスト
./gradlew lint           # Lint
```

## サブエージェントの使い分け

- **tech-architect**：アーキテクチャ・設計判断が必要なとき
- **spec-guardian**：実装・設計が仕様書と整合しているか検証するとき（読取り専用）
- **test-engineer**：テスト設計・テストケース監査
- **code-reviewer**：実装完了後の品質レビュー（読取り専用）
- 大きな実装の完了時は spec-guardian と code-reviewer のレビューを並列実行する
