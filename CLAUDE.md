# 歩継ぎ（あゆつぎ）プロジェクト

歩数連動・人生追体験型育成ゲーム。週間目標歩数を達成し続ける限り一族が代をつなぎ、未達なら家系断絶（滅亡）する。docs/ の企画書、specs/ の仕様書が正。

## 現在のフェーズ

**Webデモ開発**（歩数手入力＋デバッグ加速モード）。配布は GitHub Pages。
本番は Android（Kotlin / Jetpack Compose / Health Connect）予定。コアロジック（週間判定・人生ステージ・イベント・滅亡処理）は Android 版へ移植する前提で、入出力（歩数取得・永続化・通知）から分離して設計する。

- Webデモの実装は `web/` フォルダ。ビルド不要の静的サイト（index.html 起点）、データは localStorage。
- 技術設計の正本は `specs/05_技術設計.md`（Webデモ版とAndroid版の共通仕様／差分を定義）。
- アートスタイルの正本は `specs/06_アートスタイル仕様.md`（和紙と墨UI＋32×48ドット3頭身キャラ）。
- 公開先：https://unagisv.github.io/ayutsugi/ （GitHub Pages・gh-pages ブランチ配信）。デプロイは `tools/deploy-pages.ps1`。リポジトリは https://github.com/unagisv/ayutsugi （Public）。

## 最重要ルール：仕様書が正

- 仕様の正本は `specs/` の4部＋技術設計書。**数値・用語・判定ルールは必ず仕様書を読んで確認すること。記憶や推測で書かない。**
  - `specs/01_ゲームルール仕様.md` … 週間目標・週次判定・滅亡・SUSPENDED・グレース週
  - `specs/02_人生シミュレーション仕様.md` … 一代の長さ・ステージ・寿命計算・遺伝・家宝
  - `specs/03_人生イベント表.md` … イベント発生制御・イベント定義・世代伏線
  - `specs/04_画面仕様.md` … 5画面構成・ナビゲーション・演出
  - `specs/05_技術設計.md` … 2層構成・データモデル・Webデモ固有/Android固有の設計
- 実装中に仕様の解釈・決定が必要になったら、**先に仕様書または設計書に追記してからコードを書く**。「コードだけが知っている仕様」を作らない
- 仕様書間で記述が矛盾する場合は作業を止めてユーザーに報告する
- 仕様の不明点は specs/ を確認し、書いてなければ実装前に質問する

## 絶対に守る仕様の不変条件（詳細は必ず仕様書で確認）

- 週の定義：月曜0:00:00〜日曜23:59:59、締めは日曜24時固定（specs/01 §0-1）
- 週間目標：下限7,000／上限210,000／刻み1,000歩。変更は翌週反映・当週不可（specs/01 §0-4・§4）
- 初週はグレース週（滅亡判定対象外）、再出発時はグレースなし（specs/01 §0-6）
- 標準寿命20日・固定ステージ合計13日・老年期＝寿命−13（範囲1〜13）（specs/02主管）
- 歩数→ステータス変換・寿命・イベント達成率は**目標相対**（g＝activeGoal÷7。絶対歩数基準は廃止）（specs/02）
- 存亡判定が寿命より先。滅亡時は代替わりしない（specs/01 §7-4・§7-5）
- 判定は冪等。判定済みの週は決して覆らない。進行ポインタは単調増加（specs/01 §7-9）
- 歩数取得失敗時：再取得猶予72時間＋最終フォールバック7日、保留中はSUSPENDED（specs/01 §0-2）※Webデモは手入力のため「未入力日」に読み替え（specs/05）

## 開発ルール

- 判定ロジック（週次判定・寿命・イベント発生）は**テスト駆動**：テストを先に書く
- 時刻を扱うコードは必ず週境界（月曜0:00）・タイムゾーン・端末時刻変更を考慮し、テストに境界値ケースを含める
- 現在時刻の直接取得（`Date.now()` / `System.currentTimeMillis()` 等）をドメイン層に書かない。Clock抽象を注入してテスト可能にする（デバッグ加速モードもこのClockで実現する）
- 回答・コメント・UIテキストは日本語
- コミットは機能の区切りごとに小さく。コミットメッセージは日本語可

## サブエージェントの使い分け

- **tech-architect**：アーキテクチャ・設計判断が必要なとき
- **spec-guardian**：実装・設計が仕様書と整合しているか検証するとき（読取り専用）
- **test-engineer**：テスト設計・テストケース監査
- **code-reviewer**：実装完了後の品質レビュー（読取り専用）
- 大きな実装の完了時は spec-guardian と code-reviewer のレビューを並列実行する
