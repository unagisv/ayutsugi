## 1. Claude Code のセットアップ（Windows）

### 1-1. インストール

PowerShell を開き（Win + X → Terminal）、以下を実行：

```powershell
irm https://claude.ai/install.ps1 | iex
```

- 管理者権限は不要。自動アップデート付き
- **終わったら PowerShell を一度閉じて開き直す**（PATHが新しいウィンドウから有効になる）

Git for Windows が未導入なら入れておく（Claude Code が Bash ツールを使えるようになり安定する）：
https://git-scm.com/downloads/win （インストーラは全てデフォルトでOK）

### 1-2. 起動とログイン

```powershell
cd C:\Users\koichi.asada\Documents\ayutsugi
claude
```

初回はブラウザが開くので Claude アカウント（Pro/Max等のサブスク）でログイン。APIキーは不要。

### 1-3. 動作確認

`claude` のプロンプトに日本語でそのまま話しかける：

```
このフォルダの企画書と仕様書を読んで、プロジェクトを要約して
```

ファイルを読んで答えれば成功。

## 2. 基本操作（最初に覚えるのはこれだけ）

| 操作 | 内容 |
|---|---|
| `/init` | プロジェクトを解析して CLAUDE.md（プロジェクトの記憶ファイル）を生成 |
| `/plan` | **プランモード**：読み取り専用で計画だけ立てさせる。設計検討はまずこれ |
| Shift+Tab | モード切替（通常 ⇄ 編集自動承認）。※Windowsではプランモードが飛ばされる既知問題あり→ `/plan` を使う |
| `/clear` | 会話リセット（別の作業に移るとき） |
| `/compact` | 長くなった会話を要約して圧縮（動作が重くなったら） |
| Esc | 実行中のClaudeを中断 |
| `exit` | 終了 |

**運用の原則**

- ファイル変更やコマンド実行は許可を求められる。内容を見て承認する。慣れるまで自動承認モードは使わない
- 大きい作業の前に「まず計画を立てて」とプランモードで方針を確認 → OKなら実装させる
- こまめに `git commit` させる（「ここまでコミットして」と言えばやってくれる）

### 2-1. CLAUDE.md の準備

`/init` 実行後、生成された CLAUDE.md を確認し、以下の趣旨が入るよう追記・修正する：

```markdown
# 歩継ぎ（あゆつぎ）
歩数連動・人生追体験型育成ゲーム。docs/企画書、specs/ の仕様書が正。

## 現在のフェーズ
Webデモ開発（歩数手入力＋デバッグ加速）。配布は GitHub Pages。
本番は Android（Kotlin/Compose/Health Connect）予定。コアロジックは移植前提で設計する。

## ルール
- 仕様の不明点は specs/ を確認し、書いてなければ実装前に質問する
- 回答・コメントは日本語
- 区切りごとに git commit
```

## 3. オーケストレーション機能

Claude Code には複数のAIエージェントを並列に動かす仕組みがある。1人に全部やらせるより、役割を分けた方が速く・質が上がる場面で使う。

### 3-1. サブエージェント（標準機能・まずはこれ）

メインのClaudeが、子エージェントを生成して作業を委任し、結果だけ受け取る仕組み。**特別な設定は不要**で、プロンプトで指示するだけで使える：

```
サブエージェントを並列で使って、specs/ の4つの仕様書をそれぞれレビューし、
矛盾・未定義の数値・実装に必要なのに決まっていない事項を洗い出して
```

`/agents` で役割を固定したカスタムサブエージェント（例：レビュー専門、テスト専門）を定義することもできる（`.claude/agents/` に保存される）。

### 3-2. エージェントチーム（実験的機能）

複数のClaude Codeセッションがチームとして協調し、相互にレビュー・議論する上位機能。有効化が必要：

`%USERPROFILE%\.claude\settings.json` に追記：

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

使い方はプロンプトで依頼するだけ：

```
エージェントチームを作って技術設計をして。
構成：ゲームロジック設計担当、UI/画面設計担当、データモデル設計担当。
互いの設計の整合性をレビューし合って、最後に統合した設計書を出して
```

### 3-3. 使い分けの目安

| 場面 | 推奨 |
|---|---|
| 仕様書の一括レビュー、調査 | サブエージェント並列 |
| 技術設計（複数観点の検討＋相互レビュー） | エージェントチーム |
| 画面ごとに独立した実装 | サブエージェント or チーム |
| 同じファイルを触る連続作業、バグ修正 | 通常の単独セッション |

**注意**：並列実行はトークン消費（使用量）が単独の数倍になる。日常作業は単独、ここぞの設計・レビューでオーケストレーションを使うのがコスパが良い。

## 4. フェーズ1：技術設計

ゴール：`specs/05_技術設計.md` を作る。Webデモ版とAndroid本番版の共通部分／差分を明確にするのがポイント。

### 手順

1. `claude` を起動し `/plan` でプランモードに
2. 以下を依頼（エージェントチームを使うならここで）：

```
docs/ と specs/ を全部読んだ上で、技術設計書 specs/05_技術設計.md を作って。
含めるもの：
- Webデモ版とAndroid版の2層構成（コアロジックは共通仕様、入出力だけ差し替え）
- データモデル（一族、当主、人生ステージ、週間判定、歴史書）
- 画面遷移（specs/04_画面仕様 ベース）
- Webデモ固有：歩数手入力UI、デバッグ加速モード（1週間を数分で進める）、
  localStorage 保存、スマホブラウザ（iPhone Safari）対応
- Android固有：Health Connect連携、Room、WorkManager通知（設計だけ。実装は後）
仕様書に書かれていない数値や未決事項はリストアップして私に確認して
```

3. 提示された計画と質問に答え、設計書を生成させる
4. **自分で読んで承認する**。ここが後工程全部の土台になる

## 5. フェーズ2：Webデモ実装

### 技術スタックの推奨

- **素のHTML/CSS/JavaScript（ビルドなし・1フォルダ完結）**
  - GitHub Pages にそのまま置けて、トラブル要因が最少
  - データ保存は localStorage（友人の端末内に保存される。サーバー不要）
- ゲームの見た目が複雑になってきたら、その時点でVite等の導入をClaudeに相談すれば移行してくれる

### 必須要件（Claudeへの指示に含めること）

```
specs/05_技術設計.md に従ってWebデモを web/ フォルダに実装して。
- ビルド不要の静的サイト（index.html 起点）。GitHub Pages で公開する
- iPhone Safari で動作すること。スマホ縦画面優先のレイアウト
- 歩数は手入力（今日の歩数を入れる欄）。日付ごとに記録
- デバッグモード：画面のどこかに隠し気味に配置し、時間を加速して
  代替わり・滅亡・歴史書まで一通り体験できるようにする
- データは localStorage 保存。リセットボタンも用意
- 「ホーム画面に追加」でアプリ風に開ける程度の最低限のPWA対応（任意）
画面ごとにサブエージェントで並列実装してよいが、コアロジックは1つのモジュールに分離して
```

### 進め方のコツ

- 一気に全部作らせず、**MVP骨格（歩数入力→週間判定→代替わり→滅亡→歴史書）を先に動かす**。企画書10章のMVP定義の通り
- 動いたら自分のスマホ（Android Chrome）でも開いて確認：PC側で `ローカルで確認したい。サーバーを立てて` と頼めばプレビュー方法を案内してくれる
- 各画面ができるたびにコミット

## 6. フェーズ3：GitHub Pages で配布

1. GitHub にリポジトリ作成（**Public**。Pagesの無料公開はPublicが条件。コードを隠したい場合は有料プラン or Netlify検討）
2. Claude Code に依頼：

```
このプロジェクトをGitHubに push して、web/ フォルダを GitHub Pages で公開する手順を進めて。
gh コマンドがなければインストールから案内して
```

   手動でやる場合：リポジトリ Settings → Pages → Branch: main / フォルダ選択 → Save
3. 数分後 `https://<ユーザー名>.github.io/<リポジトリ名>/` が発行される
4. このURLを友人にLINE等で送るだけ。iPhoneは **Safariで開く → 共有 → ホーム画面に追加** でアプリのように使える

### テスト依頼時に添えると良いもの

- 遊び方3行（歩数を毎日入力／週間目標を守ると一族が続く／デバッグボタンで早送り可）
- 聞きたいこと：①ルールは直感的に分かったか ②毎日歩数を入れるのは面倒か ③滅亡の緊張感はあるか ④何日続けたくなるか
- フィードバックはLINEでもGoogleフォームでも。まとめてテキスト化しておくと次フェーズでそのままClaudeに渡せる

## 7. フェーズ4：フィードバック反映 → Android化

### 7-1. フィードバック反映

```
friends_feedback.md にテストプレイの感想をまとめた。
仕様変更が必要なもの／Webデモの修正で済むもの／Android版で対応すべきものに分類して、
仕様書への反映案を出して
```

仕様書を更新 → Webデモに反映 → 再配布、を必要なだけ回す。

### 7-2. Androidアプリ化

ゲームルールが固まったら Android 実装へ。Webデモのコアロジックが**仕様の実体**になっているので、移植指示が明確にできる：

```
android/ フォルダに Kotlin + Jetpack Compose でAndroid版を実装する。
- web/ のコアロジック（週間判定・人生ステージ・滅亡処理）と同一の挙動にする
- 歩数は Health Connect、保存は Room、通知は WorkManager（specs/05 の通り）
まず Android Studio のセットアップとプロジェクト雛形作成から案内して
```

- Android Studio のインストールと実機デバッグ（自分のAndroid端末）もClaude Codeに聞きながらでOK
- ストア公開準備（署名、Playコンソール）は別途。まずドッグフーディング（企画書ロードマップ5）

### 7-3. 歩数手入力Web版（需要があれば）

友人のフィードバックで「Webで十分・iPhoneでも使いたい」の声が強ければ、デモを本番Web版へ昇格：デバッグモード封印、データのエクスポート/インポート追加、UI磨き込み。デモがそのまま資産になる。

## 8. 困ったとき

| 症状 | 対処 |
|---|---|
| `claude` が見つからない | PowerShellを開き直す。ダメなら再インストール |
| 使用量上限に当たった | 時間を置く。`/compact` で消費を抑える。並列実行を控える |
| Claudeが暴走/間違った方向へ | Esc で中断 → 指示し直す。壊れたら `git checkout` で戻す（だからこまめにコミット） |
| 会話が長くなり精度低下 | `/clear` で仕切り直し。文脈はCLAUDE.mdと設計書が保持してくれる |
| 仕様と違う実装をされる | 「specs/◯◯の△節に従って」とファイル名で指す癖をつける |
