## 0. 確定事項（要確認事項への回答）

旧版（v1.0）で「未確定・要レビュー」としていた各論点について、ユーザーが全項目を決定した。本セクションは「論点→決定」の形で確定値のみを記す。確定済みのため本文の「暫定」表記は撤去した。

### 0-1. 週の締め曜日選択

- 論点：締め曜日を固定するか、任意曜日選択を許すか。
- **決定：日曜24時（＝月曜0:00）固定。** 週の定義は月曜0:00:00〜日曜23:59:59。任意曜日選択は将来拡張として「今後の検討」（§11）に格下げ。
- 根拠：実装単純・判定ロジック単一化。詳細は §1-1／§5。

### 0-2. Health Connect取得失敗時の猶予仕様

- 論点：取得失敗時の再取得猶予時間と最終フォールバックの上限。
- **決定：再取得猶予72時間＋最終フォールバック7日で確定。** 締め時刻から72時間以内に再取得できれば反映、7日経過しても未計測が残れば0歩確定して判定。保留中は SUSPENDED とし滅亡も存続も確定しない（無限保留防止）。
- 根拠：取得失敗のみでは滅亡させない方針と判定の無限遅延防止の両立。詳細は §6-3／§7-3。

### 0-3. データ移行・バックアップ【★方針転換：MVPに含む】

- 論点：機種変更時の引き継ぎをMVPで対応するか。
- **決定：ローカルのエクスポート/インポート機能をMVPに含める（v1.0の「MVP対象外」を撤回）。** サーバーレス（クラウド連携なし）は維持し、ファイル書き出し/読み込みで機種変更に最低限備える。判定ロジックには影響しない。
- 根拠：機種変更時のデータ消失リスクへの最低限の備え。UI配置は目標設定（設定ハブ）＝04と整合。機能仕様は §6-5。

### 0-4. 目標の上限・下限値

- 論点：週間目標歩数の設定可能範囲。
- **決定：下限7,000／上限210,000／刻み1,000歩で確定。** 詳細は §3-2。
- 根拠：下限はコミットメント装置として意味のある最小値、上限は非現実的高目標による自滅防止。

### 0-5. 危険水域の閾値

- 論点：危険水域リマインドの発火基準。
- **決定：達成率ベースを主とする。** 当週最終日（日曜）0:00時点で達成率80%未満かつ逆転不能のとき、早期警告を1回発火。詳細は §8。
- 根拠：シンプルかつ行動喚起に有効。閾値はチューニング対象。

### 0-6. 初週（オンボーディング途中の週）の扱い

- 論点：初回起動が週の途中だった場合、その週を本判定対象にするか。
- **決定：猶予週（グレース週）扱いで確定。** 初回オンボーディングの週のみ滅亡判定の対象外。翌週の月曜0:00から本判定開始。再出発時はグレースなし。詳細は §7-8／§6-4。
- 根拠：初日からの滅亡体験を避け、オンボーディングの離脱を防ぐ。

### 0-7. 滅亡後の再出発で家名・目標を引き継ぐか

- 論点：新一族を始める際の家名・週間目標の初期値。
- **決定：家名・週間目標とも前回値を初期表示し、変更可能で確定。** 詳細は §6-4。
- 根拠：再挑戦のテンポを優先。

### 0-8. 名前・性別（02主管・本書は確定値を参照）

- 論点：プレイヤー入力と自動生成の切り分け。
- **決定：家名のみプレイヤー入力。当主名・性別は自動生成（任意で改名可）。** 生成ロジックは02の主管。

### 0-9. 時間軸・寿命（02主管・本書は確定値を参照）

- 論点：標準寿命とステージ日数配分。
- **決定（02確定値）：標準寿命20日（変動±6＝14〜26日）。ステージ標準日数（合計20）＝誕生0／幼少期3／学生期3／青年期3／壮年期4／老年期7（標準）／大往生。** 固定ステージ合計（幼少〜壮年）＝13日、老年期＝最終寿命−13（下限1日・範囲1〜13）、老年期突入＝経過13日完了時。本書は §7-4／§9-2 でこの確定値を参照する。

### 0-10. 基準歩数の目標相対化【★最重要変更】

- 論点：日次更新・寿命・イベント達成率の基準点をどう取るか。
- **決定：固定8,000歩は廃止。日次目標歩数 `g＝週間目標(activeGoal)÷7` を基準/中立点に使う。** 危険水域・達成率に関わる本書の記述はすべて「週間目標基準」で一貫させる（01は週間目標が主管のため矛盾なし）。寿命・イベント達成率での適用は02/03の主管。

## 1. 基本概念と用語

本書で用いる主要用語（共通用語集に準拠）。

| 用語 | 定義 |
|---|---|
| 一族 | 継承していく家系全体。滅亡の単位。 |
| 当主 | 現在操作している1人のキャラクター。 |
| 家名 | 一族の名前（プレイヤーが命名）。表示例「山田家」。 |
| 代 | 何代目か。表示例「山田家 3代目」。 |
| 代替わり | 大往生による次代への継承（家系は継続）。 |
| 大往生 | 寿命到来による自然死。失敗ではなく一生の完結。 |
| 滅亡 | 週間目標未達による家系断絶。新一族で再出発する。 |
| 週間目標歩数 | プレイヤーが設定する1週間の目標歩数。存亡判定の基準。 |
| 歴史書 | 滅亡した一族の記録を永久保存する場所。 |
| 家系図 | 歴代当主の一覧。 |
| 歩数蓄積量 | 当代（現当主）の累計歩数。 |
| ステータス | 健康／活力／人望／財産（各0〜100の整数）。 |
| 人生ステージ | 誕生→幼少期→学生期→青年期→壮年期→老年期→大往生。 |

### 1-1. 確定しているルール上の前提

- 指標は**歩数のみ**（距離・カロリーは使わない）。
- 週の締め判定：**日曜24時（＝月曜0:00）**。
- 週の定義：**月曜0:00:00 〜 日曜23:59:59**（端末ローカルタイムゾーン基準）。
- 目標変更は**翌週から反映**、当週分は変更不可。
- **救済モードなし**（病気・怪我・出張でも救済しない）。自衛手段は「翌週からの目標引き下げ」のみ。
- 滅亡判定はローカル（Room）で実行。オフラインでも成立する。

## 2. 存亡判定の全体像

一族の存亡は、毎週の「週間目標達成判定」のみによって決まる。

```
達成（累計歩数 ≧ 週間目標歩数）  → 一族は存続。翌週へ。
未達（累計歩数 <  週間目標歩数）  → 一族は滅亡。歴史書へ記録し、翌日から新一族で再出発。
```

- 当主の寿命（大往生）は**存亡には影響しない**。寿命は一生の完結であり、代替わりして家系は続く。
- 滅亡は週単位でしか発生しない。週の途中で一族が滅びることはない（途中で目標未達が確定しても、判定は週の締めで行う）。
- 1週間に発生する判定は最大1回（その週の締め判定）。

## 3. 週間目標の設定

### 3-1. 初回提案ロジック

オンボーディング時、過去の歩数実績から無理のない初期値を提案する。

1. Health Connect から取得可能な**直近28日間（4週間）**の日次歩数を読み取る。
2. 有効な日（未計測でない日）の歩数から**1日あたりの中央値**を算出する。
   - 平均でなく中央値を用いるのは、外れ値（極端に多い/少ない日）の影響を抑えるため。
3. 提案週間目標 ＝ `1日中央値 × 7 × 0.8`（達成可能性を高めるため8掛け）。
4. 算出値を**1,000歩単位に切り捨て**、§3-2 の上限・下限にクランプする。
5. 過去データが不足（有効日数が7日未満）の場合は、デフォルト提案値 **週21,000歩**（1日3,000歩相当）を初期表示する。
6. 提案値はあくまで初期表示であり、プレイヤーは設定範囲内で自由に変更できる。

```
有効日数 >= 7 のとき：
    median = 直近28日の有効日における日次歩数の中央値
    proposed = floor(median * 7 * 0.8 / 1000) * 1000
    proposed = clamp(proposed, 7000, 210000)
有効日数 < 7 のとき：
    proposed = 21000
```

### 3-2. 設定範囲（確定・§0-4）

| 項目 | 値 |
|---|---|
| 下限 | 週7,000歩 |
| 上限 | 週210,000歩 |
| 刻み | 1,000歩単位 |

- 日次目標歩数 `g ＝ activeGoal ÷ 7` を、危険水域・達成率の基準点として用いる（固定8,000歩は廃止・§0-10）。

### 3-3. 設定UI上の制約

- 入力はスライダーまたはステッパーで、刻み・上限・下限を物理的に超えられないようにする（UI詳細は04）。
- 初回設定の確定後、その値は**翌週からの目標としてではなく、本判定が始まる最初の週の目標**として用いられる（初週の扱いは §7-8 参照）。
- 当週の目標は設定UIから直接変更できない（§4）。変更操作はすべて「翌週反映の予約」として扱う。

## 4. 週間目標の変更（翌週反映・当週変更不可）

### 4-1. 厳密な定義

- 現在進行中の週（当週）に適用される目標歩数は、その週の**月曜0:00:00時点で確定**しており、週の途中では一切変更できない。
- プレイヤーが目標値を変更する操作は、すべて「**次回の週開始時（次の月曜0:00:00）から有効になる予約**」として記録される。

### 4-2. 変更予約の仕組み

目標は2つの値で管理する。

| フィールド | 意味 |
|---|---|
| `activeGoal` | 当週に適用中の目標歩数（変更不可）。 |
| `pendingGoal` | 次週から適用される予約値（変更可能、未設定なら null）。 |

- 当週中に目標変更操作を行うと `pendingGoal` が更新される。`activeGoal` は不変。
- 週の締め処理（§5）で次週へ繰り越す際：
  - `pendingGoal` が非nullなら `activeGoal ← pendingGoal`、`pendingGoal ← null`。
  - `pendingGoal` がnullなら `activeGoal` をそのまま継続。
- UI上は「当週の目標：35,000歩（変更不可）／来週から：30,000歩（予約中）」のように両方を明示する（04と整合）。

### 4-3. いつから新目標が有効になるか

```
月曜0:00:00（週の境界）に：
  1. 当週の締め判定を実行（§5）
  2. 存続が確定したら、pendingGoal を activeGoal に昇格
  3. 新しい週の activeGoal で次週を開始
```

- 同一週内に複数回変更しても、最後に予約した `pendingGoal` のみが翌週に反映される。
- 「滅びそうになってから当週の目標を下げる」抜け道は、当週の `activeGoal` が不変であることにより構造的に塞がれている。

## 5. 週次判定ロジック

### 5-1. 週の定義（再掲）

- 1週間 ＝ 月曜0:00:00 〜 日曜23:59:59（端末ローカルタイムゾーン）。
- 週の識別子は「その週の月曜の日付（ISO週ではなく、月曜起点のローカル日付）」で管理する。
- 締め時刻 ＝ その週の翌日曜直後の月曜0:00:00（＝日曜24:00:00）。

### 5-2. 累計歩数の集計

- 当週の累計歩数 ＝ 月曜〜日曜の各日の日次歩数の合計。
- 「未計測」日は合計に含めず保留扱い（§6-3）。確定時には0歩または再取得値で確定させる。

### 5-3. 締め処理が走るタイミングと方式（両対応）

締め判定は次の2系統で確実に実行する。両者は同一の判定関数を呼び、**冪等**であること（§7-9）。

**(A) WorkManager によるスケジュール実行**
- 各週の締め時刻（月曜0:00:00）に発火する OneTime/Periodic ワーカーを登録する。
- 発火時に「未判定の確定済み週」があれば判定する。
- アプリ未起動・バックグラウンドでもOSが許す範囲で実行される（Doze等で遅延しうるため、これ単独に依存しない）。

**(B) アプリ起動時の遅延判定（catch-up）**
- アプリ起動・フォアグラウンド復帰時に、「最後に判定した週」以降で締め時刻を過ぎた未判定週がないか走査する。
- 未判定週があれば、古い週から順に判定を実行する（§7-6 の一括反映に対応）。

> WorkManagerが遅延・失敗しても、次回起動時に (B) が必ず取りこぼしを拾う。逆に (B) より先に (A) が走っても結果は同じ（冪等）。

### 5-4. 判定の中核ロジック

```
function judgeWeek(week):
    if week は判定済み:            # 冪等性（§7-9）
        return week.result

    if week に未計測日が残る:        # §6-3
        if 未計測日の再取得猶予(72h)内 かつ 最終フォールバック(7日)未到来:
            return SUSPENDED        # 判定保留。後で再試行
        else:
            未計測日を0歩として確定   # 最終フォールバック

    total = sum(week の各日の確定歩数)
    if total >= week.activeGoal:
        week.result = SURVIVE       # 存続
        昇格処理: activeGoal ← pendingGoal (あれば)  # §4-3
    else:
        week.result = PERISH        # 滅亡 → §6
    week は判定済みにする
    return week.result
```

- **達成**：累計歩数 ≧ 週間目標歩数 → 存続。
- **未達**：累計歩数 < 週間目標歩数 → 滅亡。

## 6. 滅亡処理

### 6-1. 手順

週判定で `PERISH` が確定した場合、以下を順に実行する。

```
1. 滅亡を確定する（当該一族の状態を「滅亡」にする。以後この一族では歩数を加算しない）
2. 当該一族の記録を歴史書へ永久保存する（§6-2）
3. 最長記録（最長○代）を更新判定し、更新があれば記録・表示用フラグを立てる（§6-3）
4. 「翌日から新一族で再出発」状態に遷移する（§6-4）
```

- 滅亡は週の締め（月曜0:00:00）に確定する。プレイヤーが翌週初回起動時に滅亡を知る場合も、滅亡時刻は締め時刻として記録する。
- 滅亡確定後、当該一族に対するいかなる歩数追加・目標変更も無効。

### 6-2. 歴史書への保存内容

歴史書には少なくとも以下を永久保存する（永続化フィールドの実装詳細は後続の技術設計書で定義。画面表示は04）。

- 家名
- 続いた代数（何代目まで到達したか）
- 各当主の人生ダイジェスト（ステージ、寿命の長さ、主要イベント、最終ステータス等。詳細は02/04）
- 一族の創始日と滅亡日
- 滅亡時の週間目標歩数と、未達となった週の累計歩数

### 6-3. 最長記録の更新表示

- グローバルに「これまでの最長到達代数」を保持する。
- 滅亡時、`今回の代数 > 最長記録` なら最長記録を更新し、再出発画面・ホームで「自己ベスト更新」を表示する（表示仕様は04）。
- 最長記録は次の一族の再挑戦目標として常時表示する（企画書5.6準拠）。

### 6-4. 翌日から新一族で再出発

- 滅亡確定後、プレイヤーは**翌日から**新一族で再出発できる。
  - 「翌日」＝滅亡確定日（月曜0:00:00に締めた週の翌日 ＝ その月曜）の扱い：実装上は滅亡確定と同時に新一族を開始可能とし、新一族の最初の週は**滅亡が確定した週（その月曜から始まる新しい週）**とする。
  - すなわち、滅亡で1週を失った直後の新しい週から、新一族が本判定を開始する（新一族に初週グレース §0-6 を適用するかはオンボーディング初回のみ。再出発時はグレースなしで本判定 ＝ 緊張感を維持）。
- 再出発時の初期値（§0-7 確定）：
  - 家名：前回の家名を初期表示し、任意で改名可。
  - 週間目標：前回の `activeGoal` を初期表示し、変更可。
- 再出発は1タップで開始できる導線とする（企画書リスク対策、UI詳細は04）。

### 6-5. データ移行（エクスポート/インポート）【MVP機能・§0-3】

機種変更時のデータ消失に最低限備えるため、ローカルのエクスポート/インポート機能をMVPに含める。サーバーレス（クラウド連携なし）を維持し、ファイルの書き出し/読み込みで移行する。

- **エクスポート**：現在の一族・当主・歴史書・週間目標（`activeGoal`／`pendingGoal`）・進行ポインタ・歩数蓄積などのローカル（Room）データを1ファイルに書き出す。
- **インポート**：書き出したファイルを読み込み、ローカルデータを復元する。
- **存亡判定への影響**：本機能は移行手段であり、週次判定ロジック（§5・§7）には一切影響しない。インポート後も進行ポインタ（`lastJudgedWeek`／`maxObservedTime`）は単調増加でのみ更新する（§7-9・後退させない）。
- **UI配置**：目標設定（設定ハブ）に配置する（04と整合）。ファイル形式・保存先・I/Oの実装詳細は後続の技術設計書で定義する。

## 7. 境界条件（本仕様書の肝）

すべてのケースで挙動を一意に定義する。曖昧さを残さない。

### 7-1. 週またぎ（締め時刻ちょうど・締め直後の歩数の帰属）

- 締め時刻は月曜0:00:00。**日曜23:59:59までに記録された歩数が当週**、**月曜0:00:00以降が翌週**に帰属する。
- 境界は「月曜0:00:00ちょうど＝翌週の最初の瞬間」とする（半開区間 `[月0:00, 翌月0:00)`）。
- Health Connect の歩数レコードはタイムスタンプ（または日次集計の対象日）で帰属週を決定する。締め時刻ちょうどに跨るレコードがある場合は、レコードの開始時刻が属する日で日次集計に振り分ける（日次集計の確定方法の実装詳細は後続の技術設計書に従う）。
- 締め判定で参照するのは「日曜23:59:59時点で確定した当週の日次歩数合計」であり、締め後に遡って当週へ歩数が追加されることは原則ない（未計測日の再取得 §6-3/§7-3 を除く）。

### 7-2. 端末の時刻変更（不正対策と挙動）

ローカル判定であり端末時刻に依存するため、改ざん耐性を持たせる。判定には「端末時計」だけでなく「アプリが記録した単調増加の進行ポインタ（最後に処理した週・最後に観測した時刻）」を併用する。

| 操作 | 挙動 |
|---|---|
| **過去へ巻き戻し** | アプリは「これまで観測した最大時刻（`maxObservedTime`）」を保持する。現在の端末時刻が `maxObservedTime` より過去なら、時刻巻き戻しと判定。判定の進行は巻き戻さない（既に判定済みの週を未判定に戻さない＝冪等 §7-9）。新たな歩数の帰属日も、Health Connect 側のレコード時刻を正とし、端末時計の巻き戻しでは未来分を遡及させない。 |
| **未来へ進める** | 端末時刻を進めて締めを早回しする試み。締め判定は「Health Connect が実際にその週の日次歩数を確定できているか」に依存させる。未来日付の歩数は Health Connect 側に存在しないため、前倒しした週の累計は実データのみで判定される（水増しはできない）。なお企画方針上、歩数偽装は深追いしない（企画書9）が、時刻操作で**判定を有利化（目標を下げる/未達を回避）できない**ことは保証する。 |
| **タイムゾーン変更** | 週境界はローカルタイムゾーン基準で定義する。TZ変更時は、変更後のローカル時刻で週境界を再計算する。ただし既に判定確定済みの週は再判定しない。TZ変更で同一週が二重に締まる/飛ばされる事態を防ぐため、週識別子は「確定時のローカル月曜日付」で固定し、一度確定した週は不変とする。 |

> 基本原則：**判定済みの週は決して覆らない。歩数の真値は Health Connect を正とする。端末時計は週境界の参照にのみ使い、進行ポインタは単調増加でしか動かさない。** これにより時刻操作で滅亡を回避することはできない。

### 7-3. 歩数取得失敗（権限拒否・データ欠損・未計測日・猶予判定）

| ケース | 挙動 |
|---|---|
| **権限拒否（オンボーディングで未許可）** | 歩数を取得できない。該当日はすべて「未計測」。本判定は開始しない（権限が許可されるまでオンボーディング継続）。権限が長期に許可されないことだけで滅亡はさせない。導線は04。 |
| **一時的なデータ欠損** | その日を「未計測」として保持（0歩としない）。再取得を試みる。 |
| **未計測日が週内に残る** | §6-3／下記の猶予判定に従い、判定を保留（SUSPENDED）または最終フォールバックで確定。 |

**猶予判定（確定・§0-2）**

```
週の締め時刻(月0:00)に未計測日が残る場合：
  - 締め時刻から72時間以内に、当該未計測日の歩数を再取得できれば、その値で日次を確定し週集計に反映する。
  - 締め時刻から72時間を過ぎても再取得できない、かつ
    締め時刻から7日（最終フォールバック期限）を過ぎた場合：
      → 残る未計測日を 0歩 として確定し、週判定を実行する。
  - 上記期間中、当該週の判定は SUSPENDED（保留）とし、滅亡も存続も確定しない。
```

- 保留中はホームに「歩数データ取得待ち（判定保留中）」を表示する（04）。
- 保留が解けて判定が走った結果が未達なら、その時点で滅亡を確定する（滅亡時刻は本来の締め時刻＝月曜0:00:00として記録）。

### 7-4. 目標達成と同時の寿命到来（同一タイミングの優先順位）

「週の締め（月曜0:00:00）に、当主が大往生の寿命にも到達している」ケース。優先順位を一意に定める。

```
判定順序（番号順に実行）：
  1. まず週次判定（存亡）を確定する。  ← 存亡は寿命より先
     - 達成 → 一族は存続（SURVIVE）
  2. 存続が確定した後で、当主の寿命到来を処理する。
     - 寿命到来していれば 大往生 → 代替わり（次代へ継承）
```

- 理由：存亡は一族（家系）の継続可否を決める最上位ルール。代替わりは存続している一族の内部イベントであり、存続が前提。
- したがって「達成＋寿命到来」＝**存続したうえで代替わり発生**。家系図に当代が刻まれ、次代が始まる。
- 寿命到来の判定基準（確定・02主管）：標準寿命＝20日（変動±6＝14〜26日）。ステージ配分＝誕生0／幼少3／学生3／青年3／壮年4／老年7（＝寿命−13）。固定ステージ合計（幼少〜壮年）＝13日、老年期＝最終寿命−13（下限1日）、老年期突入＝経過13日完了時。寿命到来の厳密な計算は**02の担当**。本書は「存亡を先に確定し、その後に寿命処理」という順序のみを規定する。**02との整合確認が必要。**

### 7-5. 目標未達と寿命到来が同時

```
判定順序：
  1. まず週次判定（存亡）を確定する。
     - 未達 → 一族は滅亡（PERISH）
  2. 滅亡が確定した場合、当主の寿命到来処理は実行しない。
     → 滅亡＝家系断絶であり、代替わりは発生しない。
```

- **滅亡が優先**。滅亡で家系が断絶するため、たとえ同一タイミングで寿命に達していても**大往生・代替わりは発生しない**。
- 当主はそのまま「滅亡した一族の最後の当主」として歴史書に記録される（大往生としては記録しない）。
- まとめ（§7-4 / §7-5 の優先順位）：

| 同時イベント | 結果 |
|---|---|
| 達成 ＋ 寿命到来 | 存続 → その後に代替わり発生 |
| 未達 ＋ 寿命到来 | 滅亡（家系断絶）。代替わりは発生しない |

### 7-6. 複数日アプリ未起動後の一括反映（遡及判定）

数日〜数週間アプリを開かなかった後の初回起動時。

```
起動時 catch-up（§5-3 B）：
  lastJudgedWeek 以降、締め時刻を過ぎた未判定週を「古い順」に列挙する。
  for week in 未判定週（昇順）:
      result = judgeWeek(week)          # §5-4
      if result == PERISH:
          滅亡処理（§6）を実行し、以降の週は新一族として扱う
          break して新一族の起点を設定（最初に未達となった週で家系は断絶）
      if result == SURVIVE:
          pendingGoal 昇格などを行い次週へ
      if result == SUSPENDED:
          以降の週判定を停止し、未計測の解消を待つ（§7-3）
```

- 複数週が未判定でも、**最初に未達となった週で滅亡が確定**し、それ以降の週は旧一族では存在しない（新一族の再出発がそこから始まる）。
- 例：3週ぶん未判定で、2週目が未達なら → 1週目は存続として処理、2週目で滅亡、3週目は新一族の週として扱う。
- 起動時に滅亡が判明した場合の演出・通知は04（事後に「○月○日に一族が途絶えていた」ことを丁寧に提示）。

### 7-7. 週の途中で既に目標達成済みの場合の以降の挙動

- 累計歩数が `activeGoal` に**週の途中で到達しても、その週の判定は締め時刻まで確定しない**（達成は締めで正式確定）。
- 早期達成後も歩数は加算され続け、当代の歩数蓄積量（§02のステータス・寿命・イベントに寄与）に反映される。歩き続けるインセンティブは存亡ではなく「当主の人生の充実」が担う。
- UI上は「今週は達成済み（残りは当主の人生を豊かにします）」と表示してよい（04）。ただし存続が確定するのは締め時刻。
- 早期達成week の途中で目標変更操作をしても、当週 `activeGoal` は不変（§4）。変更は翌週反映。

### 7-8. 0歩の週・アプリ初インストール直後の最初の週

- **0歩の週（権限ありで実際に1歩も歩いていない）**：累計0歩 < 目標 → 未達 → 滅亡（救済なし）。ただし「未計測（取得失敗）」と「実測0歩」は厳密に区別する（§7-3）。実測0歩は滅亡対象、未計測は保留対象。
- **アプリ初インストール直後の最初の週（初週・オンボーディング途中の週）**（§0-6 確定）：
  - 初回起動した週は**猶予週（グレース週）**として滅亡判定の対象外とする。
  - 歩数は記録・表示するが、未達でも滅亡しない。
  - 翌週の月曜0:00:00から本判定を開始する。最初に設定した `activeGoal` はこの本判定初週から有効。
  - 猶予週中の歩数も当代の歩数蓄積量には算入する（人生への反映は行う）。
  - 注意：再出発（滅亡後の新一族）にはグレース週を**適用しない**（§6-4）。グレースは初回オンボーディング時のみ。

### 7-9. 端末再起動・アプリ強制終了をまたいだ判定の冪等性

- 各週は「判定済みフラグ」と「判定結果（SURVIVE/PERISH/SUSPENDED）」を Room に永続化する。
- `judgeWeek(week)` は判定済みの週に対しては**保存済み結果を返すだけ**で副作用を持たない（冪等）。
- 滅亡処理（§6：歴史書保存・新一族生成）も「当該週の滅亡が既に記録済みか」をキーに二重実行を防ぐ。
- WorkManager と起動時catch-upが両方走っても、再起動・強制終了で処理が中断・再実行されても、結果は一意。
- 進行ポインタ（`lastJudgedWeek`, `maxObservedTime`）は単調増加でのみ更新し、後退させない。

## 8. 通知：危険水域リマインド

### 8-1. 発火条件（確定・§0-5）

- **主条件**：当週最終日（日曜）0:00時点で、当週累計歩数の達成率が **80%未満** のとき、危険水域リマインドを発火する（達成率は `当週累計 ÷ activeGoal`＝週間目標基準）。
- **補助（早期警告）**：日曜以前でも、「残り日数 × 日次目標歩数 `g`（＝activeGoal÷7、§0-10）または過去の1日平均歩数」を加味しても目標到達が**算術的に不可能**になった時点で、早めに警告を1回発火してよい。
- リマインドは1日に過剰発火させない（同一週で主リマインドは原則1回、必要に応じ最終日午後に1回追加）。具体の時刻・回数は04でチューニング。

### 8-2. 内容

- 残り必要歩数（目標 − 当週累計）。
- 締めまでの残り時間（例：「今日中（日曜24時まで）」）。
- 未達の場合に何が起こるか（「このままだと○○家が途絶えます」）。トーンは脅しすぎず、行動喚起に留める（企画方針：罰は歩かせるための緊張感）。
- 文面・表示・チャンネル設定は04の担当。**04との整合確認が必要。**

### 8-3. 関連通知（参考）

- 滅亡確定の事後通知（起動前にWorkManagerで滅亡が確定した場合）。
- 代替わり（大往生）通知。
- これらの通知設計・文面は04に従う。本書は発火の論理条件のみ規定する。
