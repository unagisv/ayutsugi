本書は『歩継ぎ』の技術設計を定義する。**ゲームルール・数値の正本はあくまで specs/01〜04** であり、本書はそれを「どう実装するか」（アーキテクチャ・データモデル・入出力・プラットフォーム差分）に専念する。仕様書に書かれていない事項を本書で決めた場合は、§9「本書で決めた事項（要ユーザー確認）」に集約する。

## 1. 全体方針：2層構成

```
┌──────────────────────────────────────────────┐
│                  コアロジック（共通仕様）                  │
│  週次判定 / 人生ステージ・寿命 / ステータス日次更新 /       │
│  イベント発生制御・分岐 / 代替わり（遺伝・家宝・伏線） /     │
│  滅亡処理・歴史書 / catch-up・冪等性 / Clock抽象          │
├──────────────┬───────────────────────────────┤
│  Webデモ版アダプタ │  Android本番版アダプタ                │
│  歩数：手入力      │  歩数：Health Connect               │
│  保存：localStorage│  保存：Room                         │
│  時刻：仮想Clock   │  時刻：システムClock                 │
│  （デバッグ加速）    │  通知：WorkManager                  │
│  通知：なし(画面内) │                                     │
└──────────────┴───────────────────────────────┘
```

- **コアロジックの仕様は両版で同一**。Webデモは「ゲームルールと体験の検証用」であり、ここで確定したコアの挙動を Android 版（Kotlin 純粋モジュール）にそのまま移植する。
- コアは**入出力に依存しない純粋関数群**として書く：歩数の出どころ（手入力/Health Connect）、保存先（localStorage/Room）、現在時刻（仮想/実時間）はすべて外から注入する。
- **Clock抽象**：コアは現在時刻を直接取得しない（`Date.now()` / `System.currentTimeMillis()` 禁止）。Webデモのデバッグ加速モードはこの Clock の差し替え（実時刻＋単調増加オフセット）だけで実現する。

### 1-1. Webデモの技術スタック

- 素の HTML / CSS / JavaScript（ES Modules）。**ビルドなし・`web/` フォルダ1つで完結**。GitHub Pages にそのまま置ける。
- データ保存は localStorage（端末内完結・サーバーレス）。
- iPhone Safari / Android Chrome のスマホ縦画面を第一対象。
- 最低限のPWA対応（manifest＋ホーム画面追加メタタグ。Service Worker は持たない＝キャッシュ事故防止）。

### 1-2. Android版の技術スタック（設計のみ・実装は後続）

- Kotlin / Jetpack Compose / Material 3。
- ドメイン層は **Android 非依存の純 Kotlin モジュール**（将来の KMP 化に備える。`android.*` 依存禁止）。
- Health Connect（歩数取得）／Room（永続化）／WorkManager（締め判定・通知）。詳細は §8。

## 3. データモデル

localStorage に保存する状態の全体（Android 版では同じ論理構造を Room のテーブルに対応させる。§8-2）。

```
GameState {
  schemaVersion: 1
  settings {
    genderedEvents: boolean      // イベント性差トグル（03 §1-5）
  }
  clock {
    offsetMs: number             // デバッグ加速の仮想時刻オフセット（単調増加のみ）
  }
  progress {
    lastJudgedWeek: string|null  // 判定済み最終週の週ID（単調増加・01 §7-9）
    maxObservedTime: number      // 観測済み最大時刻 epoch ms（単調増加・01 §7-2）
    longestGenerations: number   // 全一族の最長到達代数（01 §6-3）
    graceWeek: string|null       // グレース週の週ID（初回のみ・01 §7-8）
  }
  goal {
    activeGoal: number           // 当週適用中の週間目標（01 §4-2）
    pendingGoal: number|null     // 翌週から適用の予約値（01 §4-2）
  }
  dynasty {                      // 現在の一族（滅亡時は history へ移して新規作成）
    familyName: string
    foundedAt: number            // 創始日時 epoch ms
    generation: number           // 現当主の代数
    pastLeaders: LeaderRecord[]  // 大往生済みの歴代当主（家系図用）
    leader: Leader               // 現当主
  }
  weeks: { [weekId]: WeekRecord }   // 週ごとの記録（現一族の分）
  days:  { [dateId]: DayRecord }    // 日ごとの記録（現当主の在位期間分）
  history: HistoryEntry[]       // 歴史書（滅亡した一族の永久記録・削除不可）
}

Leader {
  name, gender ("M"|"F")        // 自動生成（02 §1-7）・改名可
  bornAt: number                // 誕生日時 epoch ms
  elapsedDays: number           // 完了した日次バッチ数（§4-1）
  stage: "誕生"|"幼少期"|"学生期"|"青年期"|"壮年期"|"老年期"
  stats { health, vitality, charm, wealth }   // 各0〜100整数（02 §5-1）
  totalSteps: number            // 歩数蓄積量S（02 §5-4）
  expectedSteps: number         // 期待累計歩数 Σg（02 §4-3。日次バッチごとに+g）
  lifespanDays: number|null     // 最終寿命（老年期突入時に確定・02 §4-2）
  traits { appearance, personality }   // 遺伝形質（02 §7-3）
  sickFlag: boolean             // 病気誘発フラグ（健康≤30・02 §5-3）
  flags {                       // 世代伏線（03 §5・MVPフル実装）
    childhoodFriend: boolean    //   E03高（幼少からの友人）
    education: "高"|"中"|"低"|null      // E04（学歴フラグ）
    job: string|null            //   E06（職業フラグ："安定"|"挑戦"|"家業"）
    married: boolean            //   E07
    spouseTalent: boolean       //   E07高（配偶者素質）
    childQuality: "高"|"中"|"低"|null   // E09（次代への遺伝の質）
    familyTradition: "堅実"|"挑戦"|null // E10（家風フラグ）
    legacyChoice: "健康"|"人脈"|"財産"|"家風"|null  // E13の選択
    connectionHeirloom: boolean //   E11高（人脈系家宝候補）
  }
  consumedEvents: string[]      // 消化済みイベントID（E14は反復可のため含めない）
  eventLog: EventLogEntry[]     // 一生のダイジェスト用（家系図・歴史書表示）
  inheritedHeirloom: Heirloom|null   // 前代から受け取った家宝（E01表示用）
}

LeaderRecord = Leader の確定スナップショット
             + { diedAt, finalLifespan, lifetimeRate, heirloomLeft: Heirloom }

Heirloom { rank: "金"|"銀"|"銅", type: "健康"|"人脈"|"財産"|"家風", bonus: 10|6|3 }

WeekRecord {
  weekId: string                // その週の月曜のローカル日付 "YYYY-MM-DD"（01 §5-1）
  activeGoal: number            // その週に適用された目標（確定後は不変）
  judged: boolean               // 冪等性フラグ（01 §7-9）
  result: "SURVIVE"|"PERISH"|"GRACE"|null
  totalSteps: number            // 判定時に確定した週累計
}

DayRecord {
  dateId: "YYYY-MM-DD"
  steps: number|null            // null＝未入力（Webデモは判定時に0歩確定・§9-D1）
  g: number                     // その日の日次目標歩数＝その週のactiveGoal÷7（02 §5-2）
  finalized: boolean            // 日次バッチ適用済みか
}

HistoryEntry {                  // 歴史書（01 §6-2）
  familyName, generations
  foundedAt, perishedAt         // 創始日・滅亡日
  goalAtPerish, totalAtPerish   // 滅亡週の目標と累計
  leaders: LeaderRecord[]       // 歴代当主ダイジェスト（最後の当主含む）
}
```

### 3-1. 不変条件（コードで保証する）

- `progress.lastJudgedWeek`・`progress.maxObservedTime`・`clock.offsetMs` は**単調増加でのみ更新**（01 §7-2／§7-9。インポート後も後退させない）。
- `WeekRecord.judged == true` の週は結果・累計とも以後不変。
- `history` は追記のみ（削除UI を作らない。企画書 5.6「永久保存」）。
- ステータスは更新のたびに 0〜100 へクランプ。`totalSteps`・`expectedSteps` は単調増加。

## 4. コアロジック処理フロー

### 4-1. 時間の進め方：日次バッチと catch-up

- 時間の最小単位は1日（02 §2-1）。**日境界＝ローカル0:00**、週境界＝月曜0:00（01 §5-1）。
- `elapsedDays`＝「完了した日次バッチの数」。当主誕生日の部分日も1日と数え、誕生後最初の0:00で1日目が完了する（§9-D2）。
- 起動時・時刻前進時に `catchUp(now)` を実行する。最後に処理した日から `now` までの**すべての日境界を古い順に**処理する（01 §5-3 B・§7-6）：

```
catchUp(now):
  for 各日境界 t（未処理分を昇順）:
    1. 日次バッチ：前日 d を確定
       - steps = days[d].steps ?? 0      # Webデモ：未入力は0歩（§9-D1）
       - ステータス日次更新（02 §5-2）／ totalSteps += steps ／ expectedSteps += g
       - elapsedDays += 1
       - elapsedDays == 13 なら最終寿命を確定（02 §4-2）
       - ステージ遷移判定（02 §3-2）
       - イベント発生判定（§4-4）
    2. t が月曜0:00 なら週次判定 judgeWeek（01 §5-4）
       - PERISH → 滅亡処理して catch-up 中断（以降は再出発待ち・01 §7-6）
       - SURVIVE/GRACE → pendingGoal 昇格（01 §4-3）
    3. 存続中で elapsedDays >= lifespanDays なら大往生 → 代替わり（§4-5）
       # 順序2→3 が 01 §7-4／§7-5 の「存亡が先・滅亡時は代替わりなし」を実現する
  maxObservedTime = max(maxObservedTime, now)
```

- すべての処理は**冪等**：判定済みの週・確定済みの日を再処理しない。再読み込み・中断・二重実行でも結果は一意（01 §7-9）。

### 4-2. 週次判定

01 §5-4 のとおり。Webデモでは歩数アダプタが常に確定値（未入力＝0歩）を返すため **SUSPENDED は発生しない**が、コアの判定関数は SUSPENDED を返せる構造を保つ（Android 版で 72h＋7日猶予を実装するため。§9-D1）。

- グレース週（`progress.graceWeek`）は未達でも `GRACE`（存続扱い・滅亡判定対象外）。初回オンボーディング週のみ設定し、再出発時は設定しない（01 §0-6／§7-8）。
- 存続時の `pendingGoal` 昇格、滅亡時の処理順（滅亡確定→歴史書→最長記録→再出発待ち）は 01 §4-3／§6 のとおり。

### 4-3. ステータス・寿命・達成率（02準拠）

- 日次更新式・初期値・クランプ・病気フラグ（健康≤30）は 02 §5 の式をそのまま実装する。
- 最終寿命は経過13日完了時に 02 §4-3 の式で確定（H・S・期待累計＝Σg）。老年期＝寿命−13（1〜13日）。
- 達成率 rate ＝ `totalSteps ÷ expectedSteps`（02 §4-3が正本。expectedSteps は日次バッチごとに当週の g を加算した Σg）。イベント分岐の高≥110%／中80〜110%／低<80% は 03 §6-2。`expectedSteps == 0` の間（誕生当日）は rate＝1.0（中）とみなす（§9-D13）。

### 4-4. イベント発生制御（03 §2.3 準拠）

- 発生タイミング：**当主の `elapsedDays` が奇数になった日次バッチ**で1回（＝約2日に1回。標準寿命20日で10回・§9-D3）。
- 選択優先順位：(a)現ステージ未消化の中核 →(b)現ステージ未消化のサブ →(c)老年期なら E14（余生・反復可）。(a)(b)が空で老年期以外なら発生なし。
- **E01** は代替わり・新一族開始の瞬間に必発（継承演出）。**E13** は大往生当日の直前に必発（家宝確定・§9-D3）。
- 発生したイベントはホームの導線→モーダルで提示し、選択肢確定時に**その時点の達成率**で高中低分岐を適用する（03 §2.3-4）。未消化のまま放置された発生済みイベントは次のイベント発生で押し出さず、選択待ちのまま保持する（同時に発生中のイベントは最大1件・§9-D14）。

### 4-5. 代替わり（02 §7 準拠）

1. 前当主を `LeaderRecord` として確定し家系図へ（生涯通算達成率 lifetimeRate を記録）。
2. 家宝確定：ランク＝lifetimeRate（金≥110%／銀80〜110%／銅<80%、必ず銅以上）。種別＝E13の選択（未選択なら argmax(健康,人望,財産)）。ステータス補正先は 02 §7-4 のとおり argmax（§9-D8）。
3. 次代生成：名前・性別を自動生成、形質50%継承、ステータス継承20%（基礎超過分・健康/人望/財産）、活力50リセット、家宝補正、E09由来の底上げ（§9-D9）、`totalSteps`・`expectedSteps`・経過日数リセット。
4. 学歴・職業・家風・人脈フラグは「親のフラグ」として次代のイベント分岐から参照できる形で保持する（03 §5）。

### 4-6. 滅亡と再出発（01 §6 準拠）

- 滅亡確定 → `HistoryEntry` を追記 → 最長記録更新判定 → 「再出発待ち」状態。
- 再出発は1タップ。家名・週間目標は前回値を初期表示・変更可（01 §0-7）。再出発時の最初の週は滅亡が確定した週（その月曜から）で、グレースなし（01 §6-4）。

## 5. Webデモ固有設計

### 5-1. 歩数の手入力

- ホームに「今日の歩数」入力欄（数値・0〜99,999）＋保存ボタン。**入力できるのは仮想的な「今日」のみ**。同日中の上書きは可、過去日の修正は不可（§9-D1）。
- 入力値は `days[today].steps` に保存。日次バッチ（翌0:00またはデバッグ前進時）で人生へ反映される。当週累計・進捗バーは入力即時に再計算して表示する。

### 5-2. デバッグ加速モード

- ホーム最下部に目立たない「dev」リンクを置き、タップでデバッグパネルを開閉（隠し気味の配置・マニュアル§5）。
- 機能：
  - **時間を進める**：「+1日」「+1週」（`clock.offsetMs` を加算→catch-up 実行）。巻き戻しボタンは置かない（単調増加の不変条件を守る）。
  - **歩数を盛る**：「今日に+2,000歩」「今日に+10,000歩」。
  - **全リセット**：localStorage を消去して初期化（確認ダイアログ付き）。
- これにより「歩数入力→週間判定→代替わり→滅亡→歴史書→再出発」を数分で一通り体験できる。
- 仮想現在時刻 `now = 実時刻 + offsetMs`。UI に仮想日時を常時表示する（デバッグ中の混乱防止）。

### 5-3. 保存（localStorage）

- キー：`ayutsugi.v1`。書き込みは状態変更のたびに全量 JSON 保存（データ量は小さい）。
- 読み込み時に `schemaVersion` を検査し、不一致なら移行処理（当面は v1 のみ）。
- **エクスポート**：全 GameState を JSON 文字列としてファイルダウンロード＋クリップボードコピーの両対応。**インポート**：テキスト貼り付け→検証→確認ダイアログ→全置換（進行ポインタは現在値より後退させない）。配置は目標設定（設定ハブ）（01 §6-5／04 §3.4）。

### 5-4. 画面構成（04 の5画面を踏襲・単一HTMLでJS切替）

| 画面 | Webデモでの差分 |
|---|---|
| ホーム | 「今日の歩数」が Health Connect 表示から**手入力欄**になる。それ以外（家名＋代数・当主ビジュアル・ステージ・週間進捗・危険水域警告・ステータス折りたたみ・イベント導線）は 04 §3.1 のとおり |
| 家系図 | 04 §3.2 のとおり（歴代当主・ダイジェスト・家宝・伏線結果・改名導線） |
| 歴史書 | 04 §3.3 のとおり（滅亡一族の永久記録・最長記録） |
| 目標設定（設定ハブ） | 04 §3.4 のとおり（当週変更不可・翌週反映予約・性差トグル・エクスポート/インポート）＋**全データリセット**ボタン（デモ用・§9-D10） |
| オンボーディング | Health Connect 権限ステップを除き 04 §3.5 を簡略化：①コンセプト説明→②家名入力→③性差選択→④初期目標設定（提案値21,000歩・§9-D6）→⑤開始 |

- 遷移・モーダル（イベント）・演出（大往生／滅亡スキップ可→歴史書→再出発1タップ）は 04 §4 のとおり。
- 当主ビジュアル：MVPは**ステージ別の人物表現＋表情3段階（活力）**を必須とし（02 §6-5）、体型5段階・服装3段階は色・装飾の簡易表現で段階的に追加する。
- 危険水域：通知は出せないため**ホームの警告帯のみ**。条件は「日曜0:00以降で達成率80%未満」（01 §8-1 主条件。逆転不能の早期警告は Android 版で実装・§9-D7）。

### 5-5. iPhone Safari 対応

- ビューポート設定・セーフエリア（`env(safe-area-inset-*)`）対応・タップ遅延なし（`touch-action`）。
- 数値入力は `inputmode="numeric"`。
- `manifest.webmanifest`＋`apple-mobile-web-app-*` メタタグで「ホーム画面に追加」に対応（Service Worker なし）。
- localStorage は Safari の ITP により**未使用7日で消去されうる**。ホーム画面追加を推奨し、画面内にエクスポートでのバックアップを促す注記を置く（§9-D11）。

## 6. テスト方針

- コアロジック（`js/core/`）はユニットテストを**先に**書く（CLAUDE.md 開発ルール）。`web/tests/tests.js` に最小アサートで実装し、Node・ブラウザ両方で実行可能にする。
- 必須テストケース：
  - 週境界：日曜23:59:59 と月曜0:00:00 の帰属（01 §7-1）／週IDの算出
  - 週次判定：達成・未達・ちょうど一致（total == goal は達成）／グレース週／pendingGoal 昇格
  - 冪等性：同じ週を二重判定しても結果不変／catch-up を二回走らせても状態不変
  - 複数週未起動の一括反映：最初の未達週で滅亡し以降は処理されない（01 §7-6）
  - 同時イベント：達成＋寿命到来→存続後に代替わり／未達＋寿命到来→代替わりなし（01 §7-4・§7-5）
  - 寿命計算：02 §4-4 の計算例6ケースを固定値で検証
  - 日次更新式：02 §5-2 の式（d のクランプ、活力減衰、財産は負にしない）
  - 家宝・遺伝：ランク閾値（110%/80%）、継承20%は基礎超過分のみ、活力は継承しない
  - イベント発生：奇数日発生、中核必発（青年期 E06/E07、壮年期 E09/E10）、老年期 E14 反復

## 7. Webデモ⇄Android 差分一覧

| 項目 | Webデモ | Android本番 |
|---|---|---|
| 歩数取得 | 手入力（当日のみ・上書き可） | Health Connect（日次集計読み取り） |
| 未計測日の扱い | 発生しない（未入力＝判定時0歩） | 72h再取得猶予＋7日フォールバック・SUSPENDED（01 §0-2） |
| 保存 | localStorage（全量JSON） | Room（§8-2のテーブル構成） |
| 締め処理の起動 | 起動時/操作時の catch-up のみ | WorkManager＋起動時 catch-up の両対応（01 §5-3） |
| 通知 | なし（ホーム警告帯のみ） | 危険水域・滅亡事後・代替わり通知（01 §8） |
| 時刻 | 仮想Clock（実時刻＋単調増加オフセット） | システムClock＋maxObservedTime改ざん対策（01 §7-2） |
| 初回目標提案 | 固定21,000歩を初期表示 | 直近28日の中央値×7×0.8（01 §3-1） |
| デバッグ加速 | あり（隠しパネル） | なし（ビルドフラグでのみ有効化） |
| コアロジック | js/core/（ES Modules） | 純Kotlinモジュール（同一仕様を移植） |

## 8. Android固有設計（設計のみ・実装は後続フェーズ）

### 8-1. モジュール構成

```
android/
├── core/          # 純Kotlin（android.* 依存なし・KMP化候補）
│   ├── time/      #   週境界・WeekId（kotlinx-datetime）
│   ├── rules/     #   確定値定数
│   ├── life/ events/ game/   # Webデモ js/core/ と同一仕様
│   └── port/      #   StepSource / GameStateRepository / Clock の interface
└── app/           # Android（Compose UI・Health Connect・Room・WorkManager）
```

### 8-2. Room テーブル（GameState の正規化）

| テーブル | 主キー | 対応 |
|---|---|---|
| `progress` | 単一行 | progress＋goal（lastJudgedWeek／maxObservedTime／activeGoal／pendingGoal／longestGenerations／graceWeek） |
| `dynasty` | id | 現一族（familyName／foundedAt／generation） |
| `leader` | id | 現当主＋flags（次世代影響フラグの保存領域を最初から確保・03 §1-4） |
| `leader_record` | id | 歴代当主スナップショット（dynastyId 外部キー） |
| `week_record` | weekId | 週の判定結果（judged は不変） |
| `day_record` | dateId | 日次歩数・g・finalized |
| `history_entry` | id | 歴史書（削除しない） |
| `event_log` | id | イベント結果ダイジェスト（leaderId 外部キー） |

### 8-3. Health Connect 連携

- `READ_STEPS` 権限。日次集計は `AggregateRequest(StepsRecord.COUNT_TOTAL)` をローカル日単位で取得。
- 取得失敗・権限喪失時は該当日を「未計測」とし、01 §7-3 の猶予判定（72h／7日／SUSPENDED）をコアの判定関数に渡す。
- 歩数の真値は Health Connect を正とし、端末時計巻き戻しでは遡及させない（01 §7-2）。

### 8-4. WorkManager

- 週締め（月曜0:00）向けの判定ワーカー＋日次バッチワーカー。Doze による遅延前提で、**起動時 catch-up と同一の冪等な判定関数**を呼ぶ（01 §5-3）。
- 通知：危険水域リマインド（日曜0:00 評価）・滅亡事後通知・代替わり通知（01 §8）。

## 9. 本書で決めた事項（仕様書に記載がないもの・要ユーザー確認）

実装を進めるため、仕様書（01〜04）に明記のない以下を本書で決定した。**いずれもコアの数値ルールを変えるものではない**が、確認のうえ異議があれば修正する。

| # | 決定事項 | 内容と根拠 |
|---|---|---|
| D1 | Webデモの未入力日は判定時に0歩確定 | 手入力には「再取得」概念がないため、SUSPENDED／72h猶予はWebデモでは発生させない（コアはSUSPENDED対応構造を維持しAndroid版で使用）。過去日の後from入力も不可（時刻不正対策の単純化） |
| D2 | 経過日数の数え方 | 「完了した日次バッチ数」。誕生日の部分日も1日と数え、誕生後最初の0:00で1日目完了。02の「累積日」をこれに対応させる |
| D3 | イベント発生日 | 経過日数が**奇数**になる日次バッチで発生（≒約2日に1回、標準寿命20日で10回）。これにより青年期（7・9日目）でE06/E07、壮年期（11・13日目）でE09/E10の中核必発が成立。E01は就任瞬間、E13は大往生当日の直前に必発 |
| D4 | 当主名の自動生成 | 男女別の和風名リストからランダム（性別は50/50）。02が主管とする生成ロジックの具体実装 |
| D5 | ステータス閾値（03 §6.3） | 02に具体値がないため**60以上**で統一：人望≥60でE07良縁選択肢+1、財産≥60でE10挑戦選択肢が有利寄り、等 |
| D6 | Webデモの初回目標提案 | 歩数履歴が存在しないため、01 §3-1のフォールバック値**21,000歩**を初期表示 |
| D7 | 危険水域の早期警告（逆転不能） | 「算術的に不可能」の定義に1日上限歩数の仮定が必要なため、Webデモでは主条件（日曜0:00・達成率80%未満の警告帯）のみ実装。早期警告はAndroid版で設計 |
| D8 | 家宝の種別と補正先 | ランク＝生涯通算達成率（02 §7-4）。種別＝E13の選択（未選択時はargmax）。**ステータス補正先は02のとおりargmax(健康,人望,財産)**とし、E13の選択は種別（次代イベントへの作用方向）のみを決める |
| D9 | E09「次代に強い遺伝・家宝の素地」の数値化 | 03に数値がないため：E09高＝次代の初期ステータス全種+2、中＝+1、低＝+0（クランプ前に加算） |
| D10 | 設定ハブに全データリセットを追加 | デモ配布でテスターが最初からやり直せるように。Android版では持たない（エクスポート/インポートのみ） |
| D11 | Safari の localStorage 7日消去対策 | ホーム画面追加の推奨とエクスポートバックアップの促しを画面内に注記（技術的回避はしない） |
| D12 | 週IDの形式 | その週の月曜のローカル日付 `YYYY-MM-DD` 文字列（01 §5-1の「月曜起点のローカル日付」の具体化） |
| D13 | 達成率の初期値 | `expectedSteps == 0`（誕生当日）の間は達成率1.0（中）とみなす（ゼロ除算回避） |
| D14 | イベントの同時発生数 | 発生中（選択待ち）イベントは最大1件。選択待ちのまま次の発生日が来た場合、新規発生はスキップする（「1日数分」の思想・04 §0 に整合） |
